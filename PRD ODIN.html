🌍 1. VISIÓN & PRINCIPIOS DEL ECOSISTEMA NÓRDICO

1.1 Declaración de Visión

Evolucionar cada herramienta del ecosistema Gentle AI en un homólogo nórdico local-first, open-source y sin dependencias de nube. Mantener la simplicidad de instalación de Gentle, pero elevar la arquitectura a nivel enterprise mediante motores vectoriales locales, políticas como código, sincronización CRDT y runtime seguro WASM.

1.2 Principios Fundamentales

Principio Implicación Técnica
🏔️ Local-First Todo funciona offline. La nube es opcional (sync/observabilidad).
🔓 100% OSS Cero licencias propietarias. Stack validado por licencia (MIT/Apache/BSD/MPL).
📦 Instalación Progresiva local (1 binario) → docker (stack completo) → cluster (K8s). Un solo comando.
🛡️ Privacidad por Defecto Cifrado en reposo, sandbox de plugins, sin telemetría oculta, keys locales.
🔄 Compatibilidad Gentle Migración automática (odyn migrate --from gentle). Sin breaking changes en v1.0.
🤝 Para la Banda Documentación clara, costos $0, gobernanza abierta, alternativas libres a todo.

🔍 3. ANÁLISIS DETALLADO POR COMPONENTE

3.1 gentle-ai → 🐺 Odin AI (Core Orchestrator)

✅ Ventajas Actuales: CLI en Go, SDD de 9 fases, integración MCP, routing básico, configuración declarativa.
⚠️ Limitaciones: Routing atado a un agente, sin fallback automático, estado volátil entre fases, sin runtime de extensiones, configuración compleja para multi-usuario.
🎯 Estrategia Local-First: Router universal con tabla dinámica por fase + fallback chain (Ollama → económico → premium). Estado persistente en ~/.odin/sessions/ con snapshots Git. WASM plugin runtime (wasmtime-go). Modo local puro: 1 binario estático, <50MB.

📦 Stack OSS: cobra (MIT), viper (MIT), wasmtime-go (Apache 2.0), go-crdt (MIT), ollama/ollama (MIT)
✅ Criterios OpenCode:

  • odin status retorna JSON/TUI válido en <800ms
  • Fallback automático en ≤3 intentos si proveedor falla
  • Plugin WASM ejecuta sin red/FS por defecto
  • odin migrate --from gentle copia configs sin romper

3.2 GGA → 🌉 Heimdall (Guardian Security)

✅ Ventajas Actuales: Pre-commit hooks, code review con LLM, switcher de proveedores, análisis básico de calidad.
⚠️ Limitaciones: Reglas hardcodeadas, dependencia de un proveedor de IA, lento en diffs grandes, sin motor de políticas, no bloquea commits inseguros.
🎯 Estrategia Local-First: Policy-as-Code con OPA (Rego) ejecutado 100% local. SAST ligero integrado: gosec + semgrep OSS. Multi-model review con fallback a modelo local. Integración nativa con pre-commit + CI (exit 0/1/2).

📦 Stack OSS: opa (Apache 2.0), semgrep (LGPLv2.1), gosec (Apache 2.0), pre-commit (MIT)
✅ Criterios OpenCode:

  • heimdall check analiza diff + aplica reglas Rego en <2s
  • Fallback a modelo local si API externa falla
  • Hook pre-commit bloquea commits con vulnerabilidades críticas
  • Reporte JSON para CI/CD pipelines

3.3 Engram → 🧠 Mimir (Memory Engine)

✅ Ventajas Actuales: SQLite FTS5, persistencia cross-session, servidor MCP, sync con Git, estructura limpia.
⚠️ Limitaciones: Solo búsqueda por keywords, sin embeddings vectoriales, contexto aislado por proyecto, sin cifrado, sin retención inteligente.
🎯 Estrategia Local-First: Dual engine: sqlite-vss (local) ↔ pgvector (docker opcional). Knowledge graph: aristas entre recuerdos. Cifrado en reposo con age (Go). Pruning automático: retiene tags arch,spec,security, descarta ruido.

📦 Stack OSS: go-sqlite3 (MIT), sqlite-vss (MIT), pgvector (PostgreSQL), age (BSD 3), go-crdt (MIT)
✅ Criterios OpenCode:

  • Búsqueda semántica <200ms en 10k registros
  • mimir encrypt/decrypt cifra/restaura sin pérdida
  • mimir sync --remote mergea sin conflictos
  • Pruning automático en background (cron/systemd)

3.4 Skills Library → 🔮 Runes (Skills Registry)

✅ Ventajas Actuales: Workflows en Markdown, contribuciones comunitarias, fácil edición, integración con SDD.
⚠️ Limitaciones: Estáticos, sin validación runtime, errores de sintaxis rompen pipeline, sin versionado, difícil descubrimiento.
🎯 Estrategia Local-First: Schema validation con CUE antes de cargar. Registro local + cache: odin rune search/install. Sandbox testing: odin rune test <id> valida output esperado. Auto-update con rollback si falla validación.

📦 Stack OSS: cue (Apache 2.0), go-yaml (MIT), wasmtime (sandbox)
✅ Criterios OpenCode:

  • odin rune install valida schema antes de escribir
  • Skill malformado no bloquea SDD, solo warning logueado
  • odin rune test retorna pass/fail con diff de salida
  • Cache local evita red para skills frecuentes

3.5 Config Sync → 🌈 Bifrost (Sync Engine)

✅ Ventajas Actuales: Sincronización cross-agent, consistencia básica, Git-backed.
⚠️ Limitaciones: Conflictos silenciosos sobrescriben configs, sin historial ni ramas, incompatible con paths WSL/macOS/Linux.
🎯 Estrategia Local-First: Repo Git bare en ~/.odin/config/. Merge CRDT para claves conflictivas. Sync multi-dispositivo: odin sync push/pull/status. Firmas GPG opcionales para commits de config.

📦 Stack OSS: go-git (Apache 2.0), go-crdt (MIT), golang.org/x/crypto/ssh (BSD)
✅ Criterios OpenCode:

  • odin sync init crea repo local en ~/.odin/config/
  • Conflictos resueltos automáticamente o marcados para revisión
  • odin sync diff muestra cambios antes de aplicar
  • Funciona en WSL2, macOS ARM, Linux x86 sin paths hardcodeados

3.6 TUI → 🕳️ Völva (Interface Engine)

✅ Ventajas Actuales: Bubbletea + Lipgloss, tema Rose Pine, interactividad básica, CLI coherente.
⚠️ Limitaciones: Tema fijo, interactividad limitada, no extensible, sin accesibilidad, sin modo script.
🎯 Estrategia Local-First: Engine de temas: völva theme set rose-pine|nord|custom.toml. Accesibilidad: alto contraste, screen reader support. Modos: --json (scripts), --quiet (CI), --interactive (default). Componentes embebibles para plugins.

📦 Stack OSS: bubbletea (MIT), lipgloss (MIT), charmbracelet/huh (MIT)
✅ Criterios OpenCode:

  • völva theme set cambia tema sin reiniciar CLI
  • --json produce salida serializable válida
  • Soporte básico screen reader + navegación por teclado
  • Plugin puede inyectar TUI component via WASM hook

3.7 E2E Pipeline → 📜 Nornir (Verification Suite)

✅ Ventajas Actuales: Matriz multi-OS, contenedores Docker, CI/CD integrado, pruebas deterministas.
⚠️ Limitaciones: Contenedores pesados, pipeline lento, no detecta tests flaky, sin benchmarks de latencia/costo.
🎯 Estrategia Local-First: testcontainers-go con contenedores efímeros + cleanup automático. Detección de flaky tests con re-ejecución + umbral de consistencia. Benchmarks locales: k6 + pprof. Reporte JSON para dashboard local.

📦 Stack OSS: testcontainers-go (MIT), k6 (Apache 2.0), go-pprof (BSD)
✅ Criterios OpenCode:

  • make test-e2e corre en <15 min en CI
  • Tests flaky marcados con @flaky y reportados
  • Benchmarks de latencia <2s para fases críticas
  • Cleanup automático de contenedores post-run

3.8 Installers → ⚒️ Dvergar (Forge/Deploy)

✅ Ventajas Actuales: curl | bash, cross-platform, setup rápido.
⚠️ Limitaciones: Sin rollback, verificación frágil, sin detección de entorno, fallo silencioso en máquinas limpias.
🎯 Estrategia Local-First: One-liner progresivo: curl -fsSL https://get.odin.ai/install.sh | bash -s -- --mode auto. Verificación criptográfica: cosign verify-blob + sigstore. Backup atómico pre-upgrade + rollback en ≤3s si falla. Detección inteligente: SO, arquitectura, Docker/Podman/K8s.

📦 Stack OSS: bash + cosign (Apache 2.0), tar + gzip, shasum
✅ Criterios OpenCode:

  • Instalación limpia en <2 min en Ubuntu/macOS/WSL2
  • cosign verify pasa antes de ejecutar
  • odin upgrade backup + migrate + rollback automático
  • odin status post-install muestra stack saludable

🏗️ 4. ARQUITECTURA CROSS-CUTTING

4.1 Estructura de Repositorio (Monorepo Inicial)

odin-ecosystem/
├── cmd/odin/              # CLI entrypoint (Odin + Völva)
├── internal/
│   ├── orchestrator/      # SDD + State + Git snapshots
│   ├── router/            # Universal model routing
│   ├── guardian/          # Heimdall (OPA + SAST)
│   ├── memory/            # Mimir (sqlite-vss + age)
│   ├── sync/              # Bifrost (go-git + CRDT)
│   ├── skills/            # Runes (CUE + sandbox)
│   ├── plugins/           # WASM runtime + hooks
│   └── verify/            # Nornir (testcontainers + k6)
├── deploy/                # Dvergar (install.sh + cosign)
├── docs/                  # Docusaurus + playgrounds
├── runes/                 # Skills registry base
├── e2e/                   # Matrix tests
├── go.work
└── Makefile

4.2 Configuración Unificada (~/.odin/config.yaml)

version: "1.0"
mode: "local"
memory:
  engine: "sqlite-vss"
  path: "~/.odin/memory.db"
  encryption: true
sync:
  backend: "git"
  remote: "git@github.com:user/odin-config.git"
guardian:
  policy_engine: "opa"
  rules: ["~/.odin/rules/"]
router:
  default: "ollama-local"
  fallback: ["openrouter", "anthropic"]
  cost_cap_daily: 0.0
observability:
  metrics_port: 9090
  log_level: "info"
plugins:
  sandbox: true
  auto_update: false

4.3 Flujo de Datos Local-First

[Usuario/Terminal] → Völva (TUI/JSON) → Odin (Orchestrator)
       ↓
   Router (decide modelo local/remoto) → Heimdall (valida políticas/SAST)
       ↓
   Mimir (indexa/consulta memoria vectorial) ↔ Bifrost (sync config CRDT)
       ↓
   Runes (carga skills validados) → Plugins (WASM sandbox)
       ↓
   Nornir (verifica E2E + benchmarks) → Dvergar (backup/upgrade atómico)
